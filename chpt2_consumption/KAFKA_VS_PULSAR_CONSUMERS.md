# Kafka vs. Pulsar: A Deep Dive into Partitioning and Consumption

This document summarizes the core differences in how Apache Kafka and Apache Pulsar handle data partitioning, consumer scaling, and message ordering. While both are powerful messaging systems, their underlying architectural philosophies lead to significantly different models, capabilities, and operational trade-offs.

## 1. The Kafka Model: Partition-Centric Consumption

In Kafka, the **partition** is the fundamental unit of parallelism. The logic for how consumers scale and where messages go is almost entirely dictated by the partition layout.

### 1.1. Partitioning Strategy: The Producer's Role

The responsibility for choosing a message's partition lies with the Producer, guided by a component called the **Partitioner**. The decision follows a strict hierarchy:

1.  **Explicit Partition:** If the message object specifies a partition, it's sent there directly.
2.  **Message Key (Most Common):** If a key is provided, Kafka hashes the key and maps it to a partition (`hash(key) % num_partitions`). This guarantees that **all messages with the same key always land in the same partition**, which is how Kafka ensures ordering for related events.
3.  **No Key:** If no key is provided, the producer uses **Sticky Partitioning**. It fills a batch with messages and sends the entire batch to a single partition. The next batch goes to the next partition, and so on. This is more efficient than a simple round-robin as it reduces network requests and broker load.

### 1.2. The Consumer Group

A Consumer Group is a set of consumer instances identified by the same `group.id`. They work together to consume a topic.

*   **Exclusive Partition Ownership:** Within a group, each partition is assigned to **exactly one** consumer. A consumer can handle multiple partitions, but a partition is never shared.
*   **Scaling Limit:** The maximum parallelism of a consumer group is **equal to the number of partitions** in the topic. If you have 10 partitions, you can have at most 10 active consumers in a group. Any additional consumers will be idle.
*   **Rebalancing:** When a consumer joins or leaves a group, Kafka triggers a "stop-the-world" rebalance to redistribute the partitions among the remaining members. This can introduce brief latency during scaling events or failures.

### 1.3. Operational Realities

*   **Partition Size:** While there's no hard limit, a partition must fit on a single broker's disk. Excessively large partitions (many TBs) are an operational risk, as they can slow down recovery and cleanup processes.
*   **Partition Imbalance:** This is a critical problem caused by a poor choice of message key (a "hot key"). One partition receives a disproportionate amount of data, overwhelming the single consumer assigned to it and causing high lag, while other consumers in the group sit idle. **Scaling out the consumer group does not fix this problem.**

## 2. The Pulsar Model: Subscription-Centric Consumption

In Pulsar, the core logical concept is the **Subscription**, a named, broker-managed entity that tracks a consumer group's progress. This architectural difference provides significantly more flexibility.

### 2.1. The Subscription: A Broker-Managed Entity

A subscription is a persistent "bookmark" on the broker. It has a name and, most importantly, a **type**. Consumers don't just join a group; they attach to a specific, named subscription.

### 3. Summary: The Power of Mixed Behaviors

The most significant advantage of Pulsar's model is the ability to have **multiple, independent subscriptions on the same topic, each with a different type.**

Consider a topic `user-activity-events`:
*   A real-time analytics dashboard can use an **`Exclusive`** subscription to get a perfectly ordered stream of all events.
*   A fleet of microservices can use a **`Key_Shared`** subscription (keyed by `user_id`) to process user data in parallel while maintaining order for each user.
*   A simple metrics counter can use a **`Shared`** subscription to burn through the data as fast as possible for simple counts, without caring about order.

All three of these use cases can run concurrently on the same topic data without interfering with each other. In Kafka, this would require more complex application-side logic, as all consumers would belong to different groups but would still be bound by the same partition-based consumption rules.
